# Backlog 課題命名ルール

このドキュメントは、法務担当者が Backlog 上で課題を起案する際の命名ルールを定義するものです。

## 1. 目的

課題名を統一する目的は以下です。

- 一覧画面で案件内容を即時に判別できるようにする
- 相手方名や契約種別で検索しやすくする
- Slack 通知、台帳、契約番号管理との整合を取りやすくする
- 手動起票と Slack 起点の課題を同じ考え方で扱えるようにする

## 2. 基本ルール

### 2.1 課題キー

Backlog の課題キーはプロジェクト標準の `LEGAL-123` 形式を前提とする。

これは Backlog 側の採番で決まるため、手入力で調整しない。

### 2.2 課題タイトル

法務担当者が手動で起票する場合の基本形は以下とする。

```text
[契約種別] 相手方名 / 案件名
```

例:

- `[NDA] 株式会社ABC / 共同企画に関する秘密保持契約`
- `[業務委託] 株式会社XYZ / キャンペーン制作業務`
- `[ライセンス] 株式会社DEF / 2026春商品化許諾`
- `[発注書] 株式会社GHI / 番組企画制作`

## 3. 入力ルール

課題タイトルは次の順序で構成する。

1. 先頭に `[契約種別]`
2. 続けて `相手方名`
3. 区切り文字として ` / `
4. 最後に `案件名` または `目的`

補足:

- `案件名` は社内で通じる案件名を短く記載する
- 契約種別は省略しない
- 相手方名は法人格を含めた正式表記を基本とする
- タイトル全体は 60 文字程度を目安にする

## 4. 契約種別ごとのテンプレート

### 4.1 NDA

```text
[NDA] 相手方名 / 案件名
```

例:

- `[NDA] 株式会社集英社 / 新規IP共同検討`
- `[NDA] 株式会社講談社 / ライセンス協議前提`

### 4.2 業務委託契約

```text
[業務委託] 相手方名 / 業務内容
```

例:

- `[業務委託] 株式会社サンプル / 動画制作業務`
- `[業務委託] 山田花子 / イラスト制作業務`

### 4.3 売買契約

```text
[売買契約] 相手方名 / 商材または案件名
```

例:

- `[売買契約] 株式会社AAA / 物販催事向け商品供給`
- `[売買契約] 株式会社BBB / 限定商品の継続販売`

### 4.4 ライセンス契約

```text
[ライセンス] 相手方名 / 権利名または案件名
```

例:

- `[ライセンス] 株式会社CCC / キャラクター商品化`
- `[ライセンス] 株式会社DDD / 配信利用許諾`

### 4.5 発注書

```text
[発注書] 相手方名 / 発注内容
```

例:

- `[発注書] 株式会社EEE / 番組企画制作`
- `[発注書] 株式会社FFF / 展示会ブース施工`

### 4.6 支払通知・検収・納品系

```text
[納品] 相手方名 / 案件名
[検収] 相手方名 / 案件名
[支払通知] 相手方名 / 案件名
```

例:

- `[納品] 株式会社GGG / 2026春販促物制作`
- `[検収] 株式会社HHH / Webサイト改修`
- `[支払通知] 株式会社III / ライセンス料 2026年3月分`

## 5. 派生課題の命名ルール

親課題に紐づく押印依頼、納品依頼、相手方確認などは、親課題キーを含めて命名する。

### 5.1 押印依頼

```text
[押印依頼] LEGAL-123 / 相手方名
```

例:

- `[押印依頼] LEGAL-123 / 株式会社ABC`

### 5.2 納品依頼

```text
[納品依頼] LEGAL-123 / 納品種別
```

例:

- `[納品依頼] LEGAL-210 / 検収書`
- `[納品依頼] LEGAL-210 / 支払通知書`

### 5.3 相手方確認

```text
[相手方確認] LEGAL-123 / 相手方名
```

例:

- `[相手方確認] LEGAL-145 / 株式会社XYZ`

## 6. 一括発注案件

一括発注は通常案件と分けて、期間や対象を明示する。

```text
[一括発注] YYYY-MM 対象内容
```

例:

- `[一括発注] 2026-03 企画発注書`
- `[一括発注] 2026-04 販促制作案件`

## 7. 記載しないもの

タイトルには次の情報を入れない。

- `至急`
- `重要`
- `確認お願いします`
- 担当者名
- ステータス名
- 長い説明文

これらは次で管理する。

- 優先度
- Backlog ステータス
- 説明欄
- カスタム属性
- コメント

## 8. NG 例

避けるべきタイトル例:

- `契約書作成`
- `ABC案件`
- `NDA`
- `法務確認`
- `至急対応`
- `株式会社ABCとの件`

理由:

- 契約種別しか分からない
- 相手方が検索しにくい
- 一覧で案件を識別できない
- 通知時に意味が弱い

## 9. 推奨運用

手動起票時は次の順序で作成する。

1. 契約種別を決める
2. 相手方正式名を確認する
3. 案件名を 20 から 30 文字程度で要約する
4. `[契約種別] 相手方名 / 案件名` の形式で入力する

## 10. システム実装との整合

現在のアプリ実装では、Slack / ワークフロー起点のタイトルは概ね次の思想で生成される。

- 押印依頼: `押印リクエスト {Backlog課題キー}`
- 納品依頼: `納品リクエスト {Backlog課題キー} {納品種別}`
- 一括発注: `企画発注書一括作成`
- 通常案件: `{ワークフロー名}: {相手方名}`

そのため、法務担当者の手動起票も「種別 + 相手方 + 案件内容」で揃えると、運用上の違和感が最も少ない。

## 11. 最終ルール

法務担当者が Backlog を手動起票する際の標準ルールは以下とする。

```text
[契約種別] 相手方名 / 案件名
```

派生課題は以下とする。

```text
[押印依頼] LEGAL-xxx / 相手方名
[納品依頼] LEGAL-xxx / 納品種別
[相手方確認] LEGAL-xxx / 相手方名
```
